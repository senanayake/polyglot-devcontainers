# Polyglot Contract Specification Design Space

## Context

The repository already implements a task contract: every maintained environment
exposes `task init / lint / test / scan / ci`. The starters catalog
(`starters/catalog.toml`) captures composition metadata. What does not yet exist
is a machine-readable contract that a downstream project can carry alongside its
code — one that CI systems, AI agents, security tools, and future governance
dashboards can consume without reading Taskfile internals.

This is the key product primitive described as the "execution contract": a
YAML/JSON file that declares runtime, tasks, expected artifacts, and policy
gates in a stable, versionable, auditable format.

## Problem Statement

How should Polyglot define a machine-readable per-project contract so that:

- CI systems can verify project compliance without human interpretation
- AI coding agents discover task entry points without parsing Taskfile.yml
- Security tools know where to find SBOMs, scan reports, and SARIF outputs
- Enterprise governance tooling can check policy gates programmatically
- The contract does not break when Taskfile.yml internals change

## Design Space Dimensions

Key variables:
- Schema richness (simple task list vs. full policy + artifacts declaration)
- Authoring burden (how hard it is to create and maintain per-project)
- Machine-readability (how easily tools can consume it)
- Stability (how often the schema changes)
- Dual use (human-readable and machine-readable simultaneously)

## Options in the Space

### Option A: Task-Only Contract (Minimal)

**Position in space:**
- Schema richness: low
- Authoring burden: very low
- Machine-readability: high
- Stability: very high
- Dual use: yes

**Characteristics:**
A minimal YAML that declares the contract version, runtime, and the task
entry points available. No policy gates, no artifact declarations.

```yaml
contractVersion: "0.1"
name: my-python-service
runtime:
  language: python
  version: "3.12"
tasks:
  - init
  - lint
  - test
  - scan
  - ci
```

- Strengths: trivially easy to adopt, low maintenance, very stable.
- Weaknesses: cannot drive policy automation or fleet compliance checking.
- Constraints: insufficient for enterprise governance use cases.

### Option B: Full Execution Contract (Recommended)

**Position in space:**
- Schema richness: high
- Authoring burden: medium (but can be generated by `polyglot init`)
- Machine-readability: high
- Stability: medium (schema will evolve)
- Dual use: yes

**Characteristics:**
A YAML file that declares runtime, task entry points, expected artifact
locations, and policy gates.

```yaml
contractVersion: "0.1"
name: java-service
runtime:
  language: java
  version: "21"
tasks:
  init: task init
  lint: task lint
  test: task test
  scan: task scan
  ci: task ci
artifacts:
  sbom: build/reports/sbom.spdx.json
  scan: .artifacts/scans/trivy-java.json
  sarif: .artifacts/scans/security.sarif
policy:
  criticalVulnerabilities: fail
  secrets: fail
  unsignedImages: warn
```

- Strengths:
  - Drives `polyglot doctor` compliance checking.
  - Enables fleet inventory tooling.
  - Agents can discover task interface and artifact locations.
  - Enterprise governance can verify policy gates programmatically.
- Weaknesses:
  - More maintenance than Option A.
  - Schema versioning requires discipline.
- Constraints:
  - Should be generated by `polyglot init`, not handwritten.

### Option C: Embedded devcontainer Metadata Only

**Position in space:**
- Schema richness: low-medium (limited to devcontainer spec extensions)
- Authoring burden: low (extends existing devcontainer.json)
- Machine-readability: high for devcontainer-aware tools, low otherwise
- Stability: tied to devcontainer spec stability
- Dual use: poor (devcontainer.json is not developer-facing prose)

**Characteristics:**
Embed Polyglot metadata as extensions inside `devcontainer.json` using the
`customizations` field.

- Strengths: no new file required, reuses devcontainer toolchain.
- Weaknesses:
  - Devcontainer spec does not have a policy or artifact model.
  - Tools that do not understand devcontainers cannot consume it.
  - Makes contract evolution coupled to devcontainer spec evolution.
- Constraints:
  - Insufficient for standalone governance use cases.

## Design Space Map

| Option | Richness | Authoring burden | Machine-readable | Stable | Dual-use |
|--------|---------|-----------------|-----------------|--------|----------|
| A: Task-only | Low | Very low | High | Very high | Yes |
| B: Full contract | High | Medium | High | Medium | Yes |
| C: devcontainer only | Low-med | Low | Partial | Med | Poor |

## Pareto Frontier

Option B is the recommendation for enterprise positioning. Option A is the
minimal viable starting point for adoption.

## Schema Design

Recommended schema location: `spec/polyglot-contract.schema.json`
Recommended per-project file name: `.polyglot-contract.yaml`
Existing per-project file: `.polyglot-starter.json` (generated at init time)

Key observation: the existing `.polyglot-starter.json` proves the project can
generate per-project metadata at init time. The contract schema extends this
pattern rather than replacing it.

Recommended schema fields:
- `contractVersion` (string, semver)
- `name` (string)
- `runtime` (object: language, version)
- `tasks` (map: verb → command)
- `artifacts` (map: type → path)
- `policy` (map: gate → action enum: fail/warn/skip)
- `features` (list of feature identifiers)
- `scenarios` (list of scenario identifiers)

## Constraints That Change the Trade-Off

- If `polyglot init` generates the contract automatically, authoring burden
  drops to near zero.
- If a JSON Schema with validation is published, enterprise teams can lint
  contracts in CI without a Polyglot tool dependency.
- If the contract is the entry point for `polyglot doctor`, the spec becomes
  self-reinforcing: every new project proves the contract format.

## Implications

- Architecture: the contract spec is the stable API surface between the open
  core and the governance tier.
- Roadmap: publish a minimal v0.1 schema early; add policy and artifact fields
  incrementally.
- Agent readiness: agents that read the contract do not need to parse
  Taskfile.yml internals.

## Recommendations

1. Start with Option A schema (task-only) in `spec/` to establish the file
   convention and tooling integration.
2. Extend to Option B (full contract) in the same schema before the killer demo.
3. Generate the contract automatically in `polyglot init / starters generate`.
4. Validate the contract in `task scan` and report via `polyglot doctor`.

## Applicability

Applies to: Phase 1 spec work, `polyglot init` generator, `polyglot doctor`,
fleet reporting, enterprise positioning.
Does not apply to: internal Taskfile structure, feature authoring.
